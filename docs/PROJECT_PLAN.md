# 医院管理系统 — 从零启动方案

> 本文档基于现有需求文档（PRODUCT_SCOPE_CONFIRMATIONS.md、USER_STORIES_EPICS.md、ROLE_PERMISSION_MATRIX.md）整理，
> 用于从零开始搭建项目前的方案确认。确认后按此分期开发。

---

## 1. 项目核心目标

构建一个**面向集团的院内信息管理系统（HIS）**，覆盖门诊、急诊、住院全流程，满足多院区统一管理。

- 为医院提供**日常业务数字化**工具（挂号、病历、处方、收费、检验检查）
- 支撑**临床决策**（病历记录、诊断、医嘱、监护数据）
- 实现**精细化管理**（药品库存、设备资产、运营报表）
- 满足**合规与安全**要求（RBAC 权限、审计日志、数据隔离）

## 2. 用户使用流程

典型的一天运营流程：

```
患者来院
  ├─ ① 建档（初诊）/ 查档（复诊）         ← 挂号员
  ├─ ② 挂号 / 取号                        ← 挂号员 / 自助机
  ├─ ③ 分诊排队                            ← 分诊护士
  ├─ ④ 医生接诊                            ← 门诊/急诊医生
  │     ├─ 书写病历 + 诊断
  │     ├─ 开立检验/检查申请
  │     ├─ 开立处方
  │     └─ 转诊 / 住院申请
  ├─ ⑤ 缴费                                ← 收费员 / 自助
  ├─ ⑥ 执行
  │     ├─ 药房取药                         ← 药师
  │     ├─ 检验科采样                       ← 医技
  │     └─ 检查科室拍片                     ← 医技
  └─ ⑦ 报告回传（医生站查阅）
```

住院流程：

```
入院登记 → 分配床位 → 医嘱开立 → 医嘱执行 → 护理记录 → 出院结算
```

后台管理流程：

```
信息科维护科室/人员/字典 → 排班 → 号源发布 → 报表查看
```

## 3. 核心页面结构

按终端划分：

| 终端 | 用户 | 核心页面 |
|------|------|---------|
| **桌面端（WPF）** | 医生、护士、挂号员、收费员、药师、医技 | 挂号工作台、门诊医生站（病历/处方/申请）、住院医生站、护士工作站、药房发药、收费窗口 |
| **管理端（Web）** | 信息科、设备科、财务、运营 | 科室人员管理、字典维护、排班管理、设备台账、报表中心、用户角色权限 |
| **大屏端** | 候诊区 | 叫号大屏（WPF 独立窗口） |

## 4. 第一版必须实现的功能（V1）

V1 定位：**核心门诊流程跑通** + **基础管理功能**。

| 模块 | 功能 | 原因 |
|------|------|------|
| ① **认证与导航** | JWT 登录、菜单路由、用户上下文 | 一切功能的基础 |
| ② **组织主数据** | 院区、科室树、人员档案 | 多院区模型的前提 |
| ③ **字典管理** | ICD/收费项目等字典 CRUD | 病历处方的依赖 |
| ④ **患者建档** | 患者主索引创建、检索、查重 | 就诊入口 |
| ⑤ **挂号** | 号表维护、窗口挂号、退号 | 门诊核心入口 |
| ⑥ **门诊医生站** | 病历书写、诊断、处方开立、检验检查申请 | 临床核心 |
| ⑦ **发药** | 门诊处方审核、发药 | 处方闭环 |
| ⑧ **收费** | 处方缴费、退费 | 财务闭环 |
| ⑨ **患者检索** | 搜索、详情 360 视图 | 日常高频操作 |
| ⑩ **用户与权限** | RBAC、角色管理、数据域隔离 | 安全基线 |

## 5. 暂时不做的功能（V2+）

| 模块 | 推迟原因 |
|------|----------|
| 住院管理（入院/床位/医嘱执行） | 依赖门诊流程稳定后再扩展 |
| 重症监护（ICU波形/评分/报警） | 需要硬件对接，独立排期 |
| 远程监护 | 需要互联网端，独立排期 |
| 预约（互联网端） | 需要患者端，独立排期 |
| 设备管理（工单/巡检/校准） | 边缘模块，不影响核心医疗流程 |
| 医保接口 | 真实联调依赖院方环境，留接口位 |
| 报表中心（复杂报表） | V1 只提供基础查询导出 |
| 护士站（体温单/护理记录） | V2 随住院一起 |
| 叫号大屏 | 简单排队信息 V1 可用文本列表替代 |

## 6. 推荐技术栈

| 层 | 技术 | 理由 |
|----|------|------|
| **后端** | ASP.NET Core 8 + C# 12 | 跨平台、性能好、国内医疗行业主流 |
| **ORM** | EF Core 8 + SQL Server | 配合 DDD 仓储模式 |
| **数据库** | SQL Server 2019+ | 医疗行业主流选择 |
| **认证** | JWT + Refresh Token | 无状态、适合多端 |
| **API 文档** | Swagger / Scalar | 前后端对接 |
| **桌面端** | WPF + CommunityToolkit.Mvvm | .NET 原生桌面，医疗行业习惯 |
| **管理端** | React / Vue 3（二选一） | 比 WPF 更适合管理后台频繁迭代 |
| **架构** | DDD（Domain-Driven Design） | 复杂业务领域的行业标准 |
| **日志** | Serilog | .NET 生态标准 |
| **映射** | Mapster | 性能优于 AutoMapper |

> 如果团队小、想快速出 V1，管理端也可以先用 WPF，砍掉 Web 端减少技术栈维护成本。

## 7. 项目目录结构


## 8. 分阶段开发计划

### 阶段一：基础设施搭建（第 1-2 周）

| 任务 | 产出 |
|------|------|
| 创建解决方案、分层项目结构 | 可编译的空项目 |
| 配置 EF Core + SQL Server 连接 | 数据库可迁移 |
| 实现 JWT 认证（登录/登出/刷新） | 登录 API + 登录页面 |
| 搭建 WPF 壳工程（导航、主题） | 带侧边栏的主窗口 |
| 基础用户种子数据 | 可登录测试账号 |

### 阶段二：主数据 + 患者（第 3-4 周）

| 任务 | 产出 |
|------|------|
| 院区/科室/人员 维护页面 | 可管理组织机构 |
| 字典管理 CRUD | 字典维护功能 |
| 患者建档 + 检索 | 患者注册与查询 |
| 患者 360 视图 | 患者详情页 |

### 阶段三：挂号 + 门诊核心（第 5-8 周）

| 任务 | 产出 |
|------|------|
| 排班号表维护 | 号源管理 |
| 窗口挂号 + 退号 | 挂号业务流程跑通 |
| 门诊医生站（病历、诊断） | 病历书写与保存 |
| 门诊处方开立 | 处方创建 |
| 检验检查申请 | 申请单开立 |
| 收费（处方缴费/退费） | 费用结算 |
| 门诊发药 | 处方发药闭环 |

### 阶段四：权限 + 安全基线（第 9 周）

| 任务 | 产出 |
|------|------|
| RBAC 用户角色权限维护 | 权限管理页面 |
| 数据域隔离（院区/科室） | 跨院区越权被拒绝 |
| 审计日志记录 | 关键操作可追溯 |
| API 异常处理中间件 | 统一错误响应格式 |

### 阶段五：打磨与修复（第 10 周）

| 任务 | 产出 |
|------|------|
| UI 一致性审查 | 符合 DESIGN.md |
| 边界情况处理 | 异常流程覆盖 |
| 性能优化 | 列表分页、查询索引 |
| 部署文档 | 部署手册 |

## 9. 每个阶段的验收标准

### 阶段一验收标准
- [ ] `dotnet build` 0 错误 0 警告
- [ ] 数据库迁移成功，种子数据可查
- [ ] 登录 API 返回合法 JWT Token
- [ ] WPF 窗口显示侧边栏导航，可切换占位页面
- [ ] 使用 `admin/admin123` 可登录成功

### 阶段二验收标准
- [ ] 可新增/编辑/停用院区和科室
- [ ] 字典类型可自定义，字典项可增删改
- [ ] 患者建档支持姓名、身份证、手机号，按身份证查重
- [ ] 患者检索支持模糊搜索，结果分页展示
- [ ] 患者 360 页展示基本信息 + 就诊历史

### 阶段三验收标准
- [ ] 排班可发布/停用，号源数量准确
- [ ] 挂号成功生成就诊记录，号源扣减
- [ ] 医生可保存病历草稿和提交终稿
- [ ] 处方开立后可在待发药列表查询
- [ ] 检验/检查申请状态可跟踪
- [ ] 缴费后处方状态变更为"已缴费"
- [ ] 发药后库存扣减，账实一致
- [ ] **完整门诊闭环流程：建档 → 挂号 → 接诊 → 开处方 → 缴费 → 发药**

### 阶段四验收标准
- [ ] 不同角色登录看到不同菜单
- [ ] A 院区用户无法查看 B 院区数据
- [ ] 病历/处方/费用的关键修改记入审计日志
- [ ] 未授权 API 返回 403

### 阶段五验收标准
- [ ] 全部页面引用统一设计令牌，视觉一致
- [ ] 表单校验覆盖必填项、格式、长度
- [ ] 列表接口响应 < 500ms（本地开发环境）
- [ ] 部署文档包含环境要求、数据库脚本执行步骤、启动方式
