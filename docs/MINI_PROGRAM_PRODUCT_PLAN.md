# 医院微信小程序 — 产品规划方案

---

## 1. 核心目标

> 让患者通过微信即可完成**预约挂号、门诊缴费、报告查询**等就医核心流程，减少排队等候时间，提升就医体验。

**一句话定位**：医院面向患者的**移动端就医服务平台**。

**三个关键价值**：
- **患者**：手机上完成挂号、缴费、查报告，少跑腿
- **医生**：减少窗口拥堵，集中精力看诊
- **医院**：提升运营效率，降低窗口人力成本

## 2. 用户使用流程

### 2.1 主流程（首次就诊 → 完成就诊）

```
┌─────────────────────────────────────────────────────────────────┐
│  打开小程序                                                      │
│    │                                                             │
│    ▼                                                             │
│  微信一键登录（获取手机号）                                        │
│    │                                                             │
│    ▼                                                             │
│  首页 → 选择「预约挂号」                                           │
│    │                                                             │
│    ▼                                                             │
│  选择院区 → 选择科室 → 查看医生排班 → 选择时段 → 确认预约          │
│    │                                                             │
│    ▼                                                             │
│  预约成功 → 收到订阅消息通知                                       │
│    │                                                             │
│    ├── 就诊当天 → 查看排队进度（实时队列号）                       │
│    │                                                             │
│    ├── 医生看诊 → 开药/开检查                                      │
│    │    │                                                         │
│    │    ▼                                                         │
│    │  收到待缴费提醒 → 在线支付（微信支付）                        │
│    │    │                                                         │
│    │    ▼                                                         │
│    │  支付完成 → 药房取药 / 去做检查                               │
│    │                                                             │
│    ├── 报告出具 → 收到通知 → 查看报告                              │
│    │                                                             │
│    └── 就诊完成 → 查看就诊记录、处方、病历                         │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 各角色与系统边界

| 角色 | 使用的系统 | 说明 |
|------|-----------|------|
 | 患者 | 微信小程序 | 预约、缴费、查报告 |
| 挂号员 | WPF 后台 | 窗口挂号（已有） |
| 医生 | WPF 后台 | 接诊、开药、写病历（已有） |
| 收费员 | WPF 后台 | 窗口收费（已有） |
| 药房 | WPF 后台 | 发药（已有） |

> 小程序只做**患者端**，后台管理功能复用现有 WPF 客户端。

## 4. 第一版（MVP）必须实现的功能

> **原则**：只做患者最痛、使用频率最高的场景——**预约挂号和排队**。

### 4.1 必须功能清单

| 模块 | 功能 | 说明 |
|------|------|------|
 | **登录** | 微信一键登录 | wx.login() 获取 openid，绑定手机号 |
| | 就诊人管理 | 添加/编辑/删除就诊人（姓名、身份证、手机号） |
| **首页** | 功能入口 | 预约挂号、排队进度、报告查询（占位） |
| | 医院信息 | 名称、地址、电话 |
| **预约挂号** | 科室列表 | 按院区展示科室，支持搜索 |
| | 医生排班 | 按日期查看医生排班、时段、剩余号源 |
| | 确认预约 | 选择就诊人，确认预约信息 |
| | 预约成功 | 展示预约详情（科室、医生、时间、排队号） |
| | 我的预约 | 按状态（待就诊/已就诊/已取消）查看预约记录 |
| | 取消预约 | 就诊前可取消预约 |
| **排队进度** | 实时队列 | 查看当前排队号、前面人数 |
| **个人中心** | 基本信息 | 头像、昵称 |
| | 就诊人管理 | 管理就诊人列表 |
| | 退出登录 | - |

### 4.2 后端依赖（复用现有 API）

| 接口 | 现有 API |
|------|---------|
 | 科室列表 | `GET /api/Department` ✅ |
| 医生排班 | `GET /api/Schedule/available` ✅ |
| 创建预约 | `POST /api/Registration` ✅ |
| 取消预约 | `PATCH /api/Registration/{id}/void` ✅ |
| 查询预约 | `GET /api/Registration/by-patient/{patientId}` ✅ |
| 排队查询 | `GET /api/Encounter/queue` ✅ |

> 新增接口：微信登录（针对小程序新增）、就诊人管理 CRUD。

## 5. 暂时不做的功能

| 功能 | 不做原因 | 计划阶段 |
|------|---------|---------|
 | 在线支付（微信支付） | 涉及商户资质申请、对账、退款等复杂流程 | 第二阶段 |
| 报告查询 | 需要对接 LIS/RIS 系统，数据源可能不在当前系统 | 第二阶段 |
| 消息推送 | 微信订阅消息需模板审核，且需支付/报告等事件触发 | 第二阶段 |
| 在线建档 | 涉及患者身份核验，风险较高，建议先支持绑定已有档案 | 第三阶段 |
| 人脸识别实名认证 | 需对接第三方服务，增加开发成本和审核复杂度 | 第三阶段 |
| 住院服务 | 住院预缴、日清单查询等，场景较复杂 | 后期规划 |
| 在线问诊 | 需要视频/图文问诊能力，属于独立产品 | 后期规划 |
| 健康资讯/科普 | 内容运营成本高，非工具型核心功能 | 暂不考虑 |

## 6. 推荐技术栈

| 层次 | 技术选型 | 选型理由 |
|------|---------|---------|
 | **前端框架** | 微信原生小程序 | 原生性能最好，API 兼容性最强，无需额外构建工具 |
| **UI 组件库** | Vant Weapp | 成熟稳定，组件丰富，医院场景够用 |
| **状态管理** | 小程序全局数据 + Storage | 场景简单，无需引入 Redux 类库 |
| **后端语言** | C# ASP.NET Core | 复用现有项目技术栈 |
| **认证方式** | JWT（登录后下发 token） | 与现有系统认证机制一致 |
| **API 通信** | wx.request 封装 | 统一拦截、异常处理、token 刷新 |
| **支付** | 微信支付 JSAPI | 微信小程序标准支付方式（第二阶段） |
| **推送** | 微信订阅消息 | 一次性订阅，用户同意后可推送（第二阶段） |
| **CI/CD** | 微信开发者工具 CI | 自动化上传代码 |

## 8. 分阶段开发计划

### 第一阶段：预约挂号（MVP）— 预估 3 周

| 周 | 任务 |
|--|------|
 | 第 1 周 | 项目搭建、UI 组件库集成、全局样式、登录流程、首页框架 |
| 第 2 周 | 科室列表、医生排班、预约确认、预约成功页 |
| 第 3 周 | 我的预约、排队进度、就诊人管理、个人中心、联调测试 |

**后端并行**：新增微信登录接口、就诊人管理 CRUD（约 3 天）

### 第二阶段：在线支付 + 报告查询 — 预估 3 周

| 周 | 任务 |
|--|------|
 | 第 1 周 | 待缴费列表、微信支付对接、支付成功/失败处理 |
| 第 2 周 | 缴费历史、报告列表、检验报告详情、检查报告详情 |
| 第 3 周 | 微信订阅消息（支付通知、报告通知）、联调测试 |

**后端并行**：微信支付统一下单/回调接口、报告查询接口（约 5 天）

### 第三阶段：就诊记录 + 优化迭代 — 预估 2 周

| 周 | 任务 |
|--|------|
 | 第 1 周 | 就诊记录列表、就诊详情（诊断、处方、病历摘要） |
| 第 2 周 | 在线建档、体验优化（骨架屏、错误处理、性能优化） |
